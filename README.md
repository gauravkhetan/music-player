# Personal Music Streaming App

A private Spotify-style music streaming app for a personal MP3 library. The app is optimized for mobile, uses Google OAuth allowlisting, stores metadata in Cloudflare D1, stores audio in Cloudflare R2, and streams audio directly from R2 with short-lived signed URLs.

## Features

- Private Google OAuth login with email allowlist
- Mobile-first dark Spotify-style UI
- Library, artists, albums, favorites, playlists, and recently played pages
- Persistent bottom audio player with play, pause, seek, volume, previous, next, shuffle, repeat, and queue
- Direct R2 audio streaming through signed URLs, without proxying MP3s through Vercel
- Fuse.js instant library search
- D1-backed favorites, playlists, and recently played tracking
- R2-to-D1 sync script and daily Vercel cron endpoint
- PWA manifest and offline shell service worker

## Tech Stack

- Next.js 15 App Router
- TypeScript

## Local Setup

Install dependencies:

```bash
npm install
```

Create local env:

```bash
cp .env.example .env.local
```

Fill in `.env.local`:

```env
AUTH_SECRET=
AUTH_GOOGLE_ID=
AUTH_GOOGLE_SECRET=
ALLOWED_EMAILS=you@example.com,another@example.com
SESSION_MAX_AGE_DAYS=30
SYNC_SECRET=

R2_PUBLIC_BASE_URL=
R2_COVERS_BASE_URL=
R2_BUCKET_NAME=music-library
R2_SONGS_PREFIX=
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=

CLOUDFLARE_ACCOUNT_ID=
CLOUDFLARE_D1_DATABASE_ID=
CLOUDFLARE_API_TOKEN=

LOCAL_SEED_PATH=data/sample-library.json
```

Start the app:

```bash
npm run dev
```

Without Cloudflare D1 env vars, the app reads from `data/sample-library.json`.

## Cloudflare D1

Create and initialize the database:

```bash
wrangler d1 create music-library
wrangler d1 execute music-library --file schema.sql
```

Or apply `schema.sql` using the app env vars:

```bash
npm run d1:schema
```

Check indexed song count:

```bash
npm run d1:count
```

## Cloudflare R2

The app supports either:

- `songs/artist/album/song.mp3`
- `artist/song.mp3` at bucket root, with `R2_SONGS_PREFIX=`

Import or refresh metadata from R2 into D1:

```bash
npm run import:r2
```

Full rebuild of generated music metadata:

```bash
npm run import:r2:reset
```

Audio playback uses signed R2 URLs generated by `/api/songs/[id]/stream-url`, so the R2 bucket does not need to be public. MP3 files are not proxied through Vercel.

## Ongoing Sync

`vercel.json` schedules this endpoint daily, which works on Vercel Hobby:

```text
/api/admin/sync-r2
```

That endpoint scans R2 and upserts new MP3 metadata into D1. Manual trigger:

```bash
curl -H "Authorization: Bearer $SYNC_SECRET" https://your-app.vercel.app/api/admin/sync-r2
```

Use `?reset=true` only when you intentionally want a full rebuild and include the `Authorization` header.

## Vercel Deployment

Set these Vercel environment variables:

```text
AUTH_SECRET
AUTH_GOOGLE_ID
AUTH_GOOGLE_SECRET
ALLOWED_EMAILS
SESSION_MAX_AGE_DAYS
SYNC_SECRET
R2_PUBLIC_BASE_URL
R2_COVERS_BASE_URL
R2_BUCKET_NAME
R2_SONGS_PREFIX
R2_ACCESS_KEY_ID
R2_SECRET_ACCESS_KEY
CLOUDFLARE_ACCOUNT_ID
CLOUDFLARE_D1_DATABASE_ID
CLOUDFLARE_API_TOKEN
LOCAL_SEED_PATH
```

Deploy with Vercel's standard Next.js preset.

This is a Next.js App Router application, not a static SPA export. Do not add a catch-all SPA rewrite for Vercel. Vercel will route App Router pages and API routes correctly. A catch-all rewrite can break `/api/*`, Auth.js callbacks, and dynamic routes.

## Google OAuth

In Google Cloud Console, configure the production callback:

```text
https://your-app.vercel.app/api/auth/callback/google
```

For local development:

```text
http://localhost:3000/api/auth/callback/google
```

Only emails listed in `ALLOWED_EMAILS` can sign in.

## QA

Run typecheck and production build:

```bash
npm run typecheck
npm run build
```

Run cloud integration QA:

```bash
npm run qa:core
```

The QA script verifies:

- D1 has indexed songs
- signed R2 audio fetch works
- generated cover URLs are not broken R2 storage URLs
- favorites can be added and removed
- playlists can be created and linked to songs
- recently played can be recorded

## PWA

The service worker caches the UI shell and metadata responses. It intentionally does not cache MP3 files.
